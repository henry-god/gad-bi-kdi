# Document Thread Workflow — NBFSA Organization
# ខ្សែស្រឡាយឯកសារ — អាជ្ញាធរសេវាហិរញ្ញវត្ថុមិនមែនធនាគារ

## 1. Core Concept: Documents Are Threads, Not Files

A government document at NBFSA is NOT a static file that moves linearly from draft to approval. It is a **living thread** — a conversation between 7 organizational levels where the document bounces up and down multiple times before reaching final acceptance.

**Average lifecycle:** 2-5 revision cycles per document before final signature.

**The thread model:**
```
Think of it like an email thread:
- L7 officer starts the thread (creates draft)
- Each level adds their reply (review, notes, approval, or bounce)
- The thread grows with every action
- The document version changes with every revision
- The final signed version is just the last snapshot in a long conversation
```

## 3. Actions Available Per Level

Each level can perform specific actions on a document thread. Not all actions are available at all levels.

### L1 — Secretary General (រដ្ឋលេខាធិការ)
- **Sign** — Final signature on approved document (only L1 can sign)
- **Approve** — Accept document as ready for signature
- **Bounce** — Return to L2 with revision notes
- **Direct** — Issue new directive downward (top-down flow)
- **Cannot:** Create, draft, or edit document content

### L2 — Deputy Secretary General (អគ្គលេខាធិការរង)
- **Approve** — Recommend to Secretary General
- **Review** — Check cross-department impact and policy alignment
- **Bounce** — Return to L3 with notes
- **Direct** — Pass directives from L1 to specific departments
- **Notes** — Add annotations to document thread
- **Cannot:** Sign, create original content

### L3 — Department Director (ប្រធាននាយកដ្ឋាន)
- **Approve** — Department-level approval
- **Review** — Policy and content review
- **Bounce** — Return to L4/L5 with revision notes
- **Assign** — Assign tasks to deputy directors and office chiefs
- **Notes** — Add review comments
- **Cannot:** Sign

### L4 — Deputy Department Director (អនុប្រធាននាយកដ្ឋាន)
- **Review** — Compliance and structural review
- **Manage** — Coordinate between offices within department
- **Assign** — Delegate specific tasks to office chiefs
- **Bounce** — Return to L5/L6 with notes
- **Notes** — Add compliance notes

### L5 — Office Chief (ប្រធានការិយាល័យ)
- **Review** — Quality review of officer's work
- **Manage** — Assign and track work within office
- **Create** — Can create documents (but usually delegates to L7)
- **Bounce** — Return to L6/L7 with corrections
- **Notes** — Add review annotations

### L6 — Deputy Office Chief (អនុប្រធានការិយាល័យ)
- **Review** — First-level quality check
- **Create** — Can create documents
- **Manage** — Day-to-day supervision of officers
- **Notes** — Add comments and suggestions
- **Bounce** — Return to L7 with corrections

### L7 — Technical Officer (មន្ត្រីជំនាញ)
- **Create** — Draft new documents from scratch
- **Draft** — Initial content creation
- **Research** — Gather data, references, attachments
- **Respond** — Revise document based on bounce-back notes
- **Resubmit** — Send revised version back up the chain
- **Cannot:** Approve, sign, or bounce to others

## 4. Document Flow Patterns

### 4.1 Bottom-Up Flow (Drafting → Approval)
The most common flow. A technical officer creates a document that needs approval from above.

```
STEP  LEVEL   ACTION          DETAIL
────  ──────  ──────────────  ──────────────────────────────────────
 1    L7      Create/Draft    Officer drafts content using template
 2    L6      Review          Deputy office chief checks quality
 3    L5      Review + Notes  Office chief reviews, may add notes
 4    L4      Review          Deputy director checks compliance
 5    L3      Approve         Department director approves
 6    L2      Review          Deputy SG reviews cross-department impact
 7    L1      Approve + Sign  Secretary General gives final signature
```

**Typical duration (no bounces):** 3-7 working days
**Reality (with bounces):** 1-4 weeks

### 4.2 Top-Down Flow (Directive → Execution)
A directive from the Secretary General flows down for implementation.

```
STEP  LEVEL   ACTION          DETAIL
────  ──────  ──────────────  ──────────────────────────────────────
 1    L1      Direct          SG issues directive/instruction
 2    L2      Direct + Notes  Deputy SG adds implementation guidance
 3    L3      Assign          Director assigns to relevant offices
 4    L4      Manage          Deputy director breaks into tasks
 5    L5      Assign          Office chief distributes to officers
 6    L6      Manage          Deputy chief supervises execution
 7    L7      Research/Create Officer executes and creates response
 ─── then flows back UP ───
 8    L7      Create          Creates execution report
 9    L5→L3   Review chain    Each level consolidates
10    L2      Review          Deputy SG reviews final report
11    L1      Approve         SG accepts execution report
```

**Typical duration:** 1-3 weeks for full cycle

### 4.3 Bounce-Back Flow (Revision Cycles)
The realistic flow — documents get sent back for revision 2-5 times.

```
BOUNCE TRIGGERS (reasons a document gets sent back down):
─────────────────────────────────────────────────────────
• Wrong reference number or format
• Missing required attachment
• Incorrect budget code or amount
• Policy/regulation citation error
• Tone/wording not appropriate for recipient level
• Missing approval from parallel department
• Factual error in content
• Incomplete information
• Does not align with current directive
• Formatting/template violation
```

**Example: Budget Request Letter (realistic timeline)**
```
Day 1   L7  Drafts budget request
Day 1   L6  Quick review, passes up
Day 2   L5  Reviews, passes up
Day 2   L4  Reviews, passes up
Day 3   L3  BOUNCE → wrong budget code
Day 3   L4  Passes bounce note down
Day 3   L5  Passes to officer
Day 4   L7  Fixes budget code, resubmits
Day 4   L5  Confirms fix, sends up
Day 5   L4  Confirmed, sends up
Day 5   L3  Approves, sends to deputy SG
Day 6   L2  BOUNCE → missing cost breakdown attachment
Day 6   L3  Relays to deputy director
Day 6   L4  Assigns cost breakdown to officer
Day 7   L7  Creates cost breakdown, resubmits
Day 7   L5  Reviews attachment
Day 8   L4  Confirms complete package
Day 8   L3  Re-submits with attachment
Day 9   L2  Approves, recommends to SG
Day 10  L1  BOUNCE → adjust wording in paragraph 2
Day 10  L2  Passes revision note down
Day 10  L3  Assigns wording revision
Day 11  L7  Revises wording, resubmits
Day 11  L5→L2  Quick reviews all the way up
Day 12  L1  APPROVED + SIGNED

Total: 12 working days, 3 bounce cycles, 7 versions
```

### 4.4 Parallel Flow (Multi-Department Coordination)
Some documents require input or approval from multiple departments simultaneously.

```
                    L1 (Secretary General)
                          │
                    L2 (Deputy SG)
                    ╱     │     ╲
            Dept A     Dept B     Dept C
            L3─L7      L3─L7      L3─L7
            (draft)    (review)   (input)
                    ╲     │     ╱
                    L2 (consolidates)
                          │
                    L1 (approves)
```

## 5. Document Thread Data Model

### Thread Structure
Each document is a thread containing an ordered list of actions:

```
DocumentThread {
  id: UUID
  template_id: string              // which template was used
  title: string                    // document subject
  title_km: string                 // Khmer title
  
  // Current state
  current_level: 1-7               // where it sits now
  current_holder_id: UUID          // who has it
  direction: "up" | "down"         // ascending or descending
  status: ThreadStatus             // see below
  bounce_count: int                // times sent back
  current_version: int             // latest version number
  
  // Metadata
  created_by: UUID                 // L7 officer who started it
  created_at: datetime
  updated_at: datetime
  target_signer: UUID              // L1/L2 who should sign
  priority: "normal" | "urgent" | "confidential"
  deadline: datetime?              // if time-sensitive
  
  // Content
  latest_content: JSON             // current document data
  attachments: Attachment[]        // supporting files
  
  // History
  actions: ThreadAction[]          // full action log (see below)
  versions: ThreadVersion[]        // content snapshots per revision
}
```

### Thread Status (State Machine)
```
CREATED          → L7 drafted, not yet submitted
SUBMITTED        → Sent up, in transit
IN_REVIEW        → Being reviewed at current level
APPROVED_LEVEL   → Approved at current level, moving to next
BOUNCED          → Sent back for revision
REVISION         → Officer is revising based on bounce notes
RESUBMITTED      → Revised version sent back up
PENDING_SIGN     → Reached L1, awaiting signature
SIGNED           → L1 has signed — document is official
ARCHIVED         → Stored permanently, no further changes
CANCELLED        → Withdrawn by creator or admin
```

### Thread Action Log
Every action by every person is recorded:

```
ThreadAction {
  id: UUID
  thread_id: UUID
  actor_id: UUID                   // who performed this action
  actor_level: 1-7                 // their level at time of action
  actor_title_km: string           // their Khmer title
  
  action: ActionType               // see below
  direction: "up" | "down" | "lateral"
  
  from_level: int                  // level before this action
  to_level: int                    // level after this action
  
  notes: string?                   // comments, revision instructions
  notes_km: string?                // Khmer notes
  
  version_before: int              // document version before action
  version_after: int               // document version after action
  
  created_at: datetime
  time_held: duration              // how long this person held the doc
}
```

### Action Types
```
CREATE           → New document drafted
SUBMIT           → Sent up to next level
REVIEW           → Reviewed at current level (no decision yet)
APPROVE          → Approved at current level, forwarded up
BOUNCE           → Sent back down with revision notes
REVISE           → Content modified based on bounce notes
RESUBMIT         → Revised version sent back up
SIGN             → Final signature by L1
ANNOTATE         → Notes added without changing status
ASSIGN           → Delegated to specific person
FORWARD          → Passed laterally to another department
RECALL           → Creator pulls back before approval
ARCHIVE          → Moved to permanent archive
```

### Version Tracking
Each revision creates a new version snapshot:

```
ThreadVersion {
  id: UUID
  thread_id: UUID
  version_number: int              // 1, 2, 3, ...
  content_snapshot: JSON           // full document data at this point
  change_summary: string           // what changed
  change_summary_km: string
  triggered_by: UUID               // which action caused this version
  diff_from_previous: JSON?        // what fields changed
  created_at: datetime
}
```

## 7. How KGD System Reduces Bounces

The #1 value of the KGD system is **reducing bounce-back cycles** from 2-5 down to 0-1.

### Problem → System Solution

| Bounce Reason | Frequency | KGD Prevention |
|---------------|-----------|---------------|
| Wrong format/template | 30% | Locked templates — format cannot be wrong |
| Missing required fields | 25% | Validation — system blocks submission if fields empty |
| Wrong reference number | 15% | Auto-generated ref numbers per ministry scheme |
| Incorrect date format | 10% | Auto-formatted Khmer dates |
| Wrong tone/wording | 10% | Knowledge rules enforce tone per recipient level |
| Missing attachment | 5% | Checklist — system warns if expected attachments missing |
| Policy citation error | 5% | Knowledge base auto-suggests correct citations |

### Estimated Impact
```
Before KGD:
  Avg bounces per document: 3
  Avg completion time: 12 working days
  Officer time per document: 6 hours (including revisions)

After KGD:
  Avg bounces per document: 0.5 (most caught at creation time)
  Avg completion time: 3 working days
  Officer time per document: 1 hour (AI-assisted creation)

Savings per document: ~5 hours officer time + 9 calendar days
```

## 8. Thread-Aware System Architecture Changes

The original 6-layer blueprint needs these additions for thread support:

### New Module: Thread Engine
```
src/backend/services/thread-engine.ts

Responsibilities:
  - Create new thread (L7 creates document)
  - Submit thread upward (send to next level)
  - Bounce thread downward (return with notes)
  - Track current holder and level
  - Enforce action permissions per level
  - Record every action in action log
  - Create version snapshot on every revision
  - Calculate SLA timers and overdue alerts
  - Handle parallel department routing
```

### New Module: Inbox Service
```
src/backend/services/inbox-service.ts

Responsibilities:
  - Get pending items for a user (filtered by their level + role)
  - Get bounce-backs for a user
  - Get sent items and their current status
  - Get overdue items and alerts
  - Get department/organization statistics
  - Real-time updates via WebSocket or polling
```

### New Module: Notification Service
```
src/backend/services/notification-service.ts

Responsibilities:
  - Notify next level when document arrives
  - Notify creator when document is bounced
  - Alert on overdue items (configurable thresholds)
  - Daily digest: pending items summary per user
  - Escalation: auto-notify L3+ if stuck > N days
```

### Database Additions (Prisma)
```
model DocumentThread { ... }      // see data model above
model ThreadAction { ... }        // action log
model ThreadVersion { ... }       // version snapshots
model ThreadAssignment { ... }    // who is assigned at each level
model Notification { ... }        // notification queue
```

### API Additions
```
POST   /api/threads                    Create new thread
GET    /api/threads/:id                Get thread with full action log
POST   /api/threads/:id/submit        Submit to next level
POST   /api/threads/:id/approve       Approve at current level
POST   /api/threads/:id/bounce        Bounce back with notes

GET    /api/threads/:id/timeline       Get visual timeline of actions
```

## 9. SLA Rules (Service Level Agreement)

Default time limits per level before escalation alerts:

| Level | Max Hold Time | Escalation To |
|-------|--------------|---------------|
| L7 (drafting) | 3 working days | L5 office chief |
| L6 (review) | 1 working day | L5 office chief |
| L5 (review) | 2 working days | L3 department director |
| L4 (review) | 2 working days | L3 department director |
| L3 (approval) | 3 working days | L2 deputy SG |
| L2 (review) | 3 working days | L1 secretary general |
| L1 (signature) | 5 working days | Auto-reminder only |

**Priority override:**
- Urgent documents: all timers halved
- Confidential documents: L1/L2 timers only, no escalation
