# V5-M7.1 — Firebase Storage for DOCX Blobs (SHIPPED)

**Date:** 2026-04-17
**Commit:** (pending)

## Problem

Generated DOCX files were saved to `/tmp/kgd-storage/documents/` inside
the Cloud Function container. Cloud Functions v2 containers are ephemeral —
`/tmp` is wiped on cold starts and between invocations routed to different
instances. This meant:

- A document generated on instance A could not be downloaded if the next
  request hit instance B.
- All generated files were lost after ~15 minutes of inactivity (cold start).

## Solution

Switched the production `STORAGE_BACKEND` from `local` (filesystem) to
`firebase` (Firebase Storage bucket `gad-bi-kdi.firebasestorage.app`).

### Changes

| File | Change |
|------|--------|
| `functions/index.ts` | Set `STORAGE_BACKEND=firebase` as default env var |
| `src/backend/services/storage-service.ts` | Rewrote `getFirebaseBucket()` to reuse the admin SDK default app (ADC in Cloud Functions) instead of requiring `FIREBASE_SERVICE_ACCOUNT_JSON` |
| `src/backend/services/storage-service.ts` | `STORAGE_ROOT` now respects the env var for local fallback path |
| `storage.rules` | Already existed — server-only writes, auth-required reads |

### How it works now

```
POST /api/documents/generate
  → template-engine.ts generates DOCX buffer
  → storage-service.ts (firebase backend) uploads to:
      gs://gad-bi-kdi.firebasestorage.app/documents/{docId}.docx
  → document-service.ts stores outputFilePath = "firebase://gad-bi-kdi.firebasestorage.app/documents/{docId}.docx"

GET /api/documents/:id/download
  → storage-service.ts downloads buffer from Firebase Storage
  → streams to client as .docx attachment
```

### Local dev

Local dev still defaults to `STORAGE_BACKEND=local` (filesystem) unless
you explicitly set `STORAGE_BACKEND=firebase` with ADC credentials.

## Deploy artifacts

- Firebase Storage rules deployed (unchanged)
- Firebase Hosting redeployed with latest build
